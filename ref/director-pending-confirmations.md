## 🔴 미해결

### 1. link 충돌 정책 = fail-closed 409인가, "기존 고객으로 유도" UX인가

**두 배치 연속 등장.**

- 0706 배치 2 · PR #179 — **정방향**: 들어오는 앱 계정이 이미 다른 고객에 연결됨 → `ConflictError` → 409. *답 못 받음.*
- 0709 배치 3 · PR #205 — **역방향**: 대상 고객이 이미 다른 앱 계정에 연결됨 → 정방향과 같은 정책으로 맞춰 409 채택.

**현재 동작**: 앱 견적요청/상담신청 인박스에서 "연결"을 누르면 충돌 시 **409 에러**. 상담사는 왜 막혔는지 모른다.

**대안**: `이 앱 계정은 이미 홍길동(CU-2606-0012) 고객에 연결돼 있습니다 — 그 고객으로 이동할까요?` 유도.

**주의**: 정책이 정해지면 `quote-requests`·`consultations` **양쪽 동시 수정**(안 그러면 새 드리프트).

### 2. AI 리포트에 뜨는 고객 집합이 경계에서 바뀐다 (0709 배치 3 · PR #204)

`stale_customers`·`delivery_risk`의 무활동 일수를 `floor(경과/24h)` → **KST 달력일 차**로 통일했다(고객 목록의 관리 상태 배지와 같은 지표).

- 예: 마지막 활동 7일 전 13:00, 지금 12:00(경과 6일 23시간) → 전에는 6일이라 리포트에서 빠졌고, 목록 배지는 7일이라 "확인필요"였다. 이제 둘 다 7일.
- 결과: 경계에서 리포트에 **뜨는 고객이 늘거나 준다.** 항상 목록 배지와 **일치하는 방향**이다.

**확인 포인트**: 목록 배지와 AI 리포트가 같은 고객을 가리키는 게 의도인가.

### 3. 목록 관리 상태 배지가 견적 활동을 인정한다 (0706 배치 2 · PR #180)

활동 파생을 자식 5테이블 합집합(memos / tasks / schedules / **documents** / **quotes**)으로 통일했다. 목록 배지가 견적 작성을, AI 도구가 서류 업로드를 "담당자 활동"으로 인정하게 됐다.

**덜 stale한 방향으로만** 변한다(고객이 "정상"으로 보이기 쉬워짐).

### 4. 재문의·수동 상태 오버라이드를 AI 도구가 모른다 (0709 감사에서 발견, 미착수)

고객 목록 배지는 `재문의`(recontacted)와 워크플로우 카드의 수동 상태(`manageStatusOverride`)가 stale 버킷보다 **우선**한다. 그런데 업무 AI `stale_customers`에는 그 개념이 없다.

**결과**: 30일 넘게 방치된 재문의 고객이 목록엔 **"재문의"**, AI 리포트엔 **"장기방치"**로 나온다.

어느 쪽이 옳은지가 도메인 판단이라 코드를 손대지 않았다.

### 5. AI 힌트 목업 20문장(이사님 2026-05-19 설계 문장) → AI 생성값 교체 (2026-07-12, 사후 공유)

고객 목록 AI 힌트 말풍선이 **이름만 AI인 하드코딩**(목업 20명 전용 문장 테이블 + 미구현 `ai_summary` 폴백 — 앱 유입 고객은 빈 보라 말풍선)이었던 것을 실 생성으로 교체했다.

- **원문 보존**: 이사님이 쓴 목업 문장 원문은 git 이력에 있고, 그 문장들을 **역설계한 스펙**(단계별 관점·60자·굵게 1~2곳·평서 압축 종결)이 생성 프롬프트의 few-shot으로 톤을 보존한다.
- **동작**: 고객 데이터 실변경 시 자동 재생성(백그라운드), 값 없으면 힌트 버튼 자체를 숨김.
- **확인 포인트**: 생성 문장 톤이 이사님 기준에 맞는가 — 목록에서 몇 명 hover해 보시고 어긋나는 문장이 있으면 프롬프트를 조정한다(전 고객 재생성은 1분·비용 무시 가능).
