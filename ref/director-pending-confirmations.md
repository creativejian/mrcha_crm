### 항목 11 — 인박스 이름 매칭 제안 (#282, 2026-07-18): 🟡 **유슨생 브레인스토밍·결정·머지 — 사후 공유**

인박스("상담 신청 DB"·"앱 견적요청")의 기존 고객 매칭이 `app_user_id`→`phone`만 봐서, 수기 고객과 앱 유저가 **같은 사람인데 번호가 다르면** "신규(미연결)"로 떠 [고객 생성] 시 중복 고객이 생기던 것을 예방. 이사님이 봐야 할 사용자 가시 변화 1건:

1. **인박스 "신규(미연결)" 행에 제안이 뜬다**: 같은 이름의 앱 미연결 고객이 있으면 [고객 생성] 위에 "이름이 같은 미연결 고객: 김지운 CU-… [연결]"이 노출된다. 상담사가 [연결]을 누르면 앱 계정이 그 **기존 수기 고객**에 붙어 메모·할일·견적이 보존된다(새 중복 대신 기존 고객 재사용). 동명이인이면 후보를 여러 개 보여주고 상담사가 고른다(자동 연결 아님). [고객 생성](새로 생성)도 그대로 남는다.

방식 결정(유슨생 2026-07-18): **예방만**(이미 생긴 중복 레코드를 합치는 사후 병합은 실측 중복 0건이라 보류 — spec §6). droppedPhone 선택 UI도 현행 토스트 유지. 되돌리려면 제안 UI 렌더 제거(파생 필드는 무해 잔존).

### 항목 10 — 고객 전화번호 소유권 분리 (#276, 2026-07-17): 🟡 **유슨생 논의·확정·머지 — 사후 공유**

유슨생과 실시간 논의로 확정(앱 정책 전제 — 휴대폰 인증 강제·상담 로그인 필수·등록 번호로만 상담 — 는 유슨생이 확인). 이사님이 봐야 할 사용자 가시 변화 4건:

1. **연락처 2칸이 됐다**: 주 번호(앱 연결 고객 = 앱 등록 번호 자동 표시·수정 불가 / 수기 고객 = 기존처럼 편집) + **추가 연락처**(누구나 편집 — "다른 번호로 연락달라" 케이스, 라벨 없음). 목록 연락처 줄에도 병기·검색 됨. **목록 표시 = 이사님과 유슨생이 상의해 "추가 번호도 목록 노출"로 결정(2026-07-17)**.
2. **앱 연결 고객의 번호가 항상 앱 최신값**: 저장 스냅샷 폐기 → 조회 시 `profiles.phone_number` 합성. 고객이 앱에서 번호를 바꾸면 CRM도 자동 반영. (기존 실측: 제임스 = 더미 번호 표시 중이었음 → 실번호로 교정, 김지안 = 공란 → 실번호 표시.)
3. **승격·연결이 CRM 연락처 칸을 채우지 않는다**(구 "폼 우선"·"빈 연락처 보강" 폐기 — 과거 상담 폼 번호는 실측상 전부 테스트 노이즈). 연결 시 상담사가 적어둔 기존 번호는 앱 번호와 다르면 **추가 연락처로 자동 보존**.
4. **전화 매칭(동일인 추정) 후보 = 앱 미연결 고객만** — 이미 연결된 고객이 가족 공유 번호로 다른 유저의 후보에 뜨던 것 소멸(김지운·김지안↔김민준 실측 케이스 축소).

기각/조정 지점: 목록 병기 형태(한 줄 `주 · 추가`)·추가 연락처 라벨 없음(칸 하나) — 어휘·배치는 UI 폴리시로 조정 가능. 데이터 모델(불변식·read-through)은 되돌리려면 마이그 원복 필요.

### 항목 8 — 수동 관리 상태의 AI 리포트 멤버십: 🟡 **① 구현 완료(#235) — 사후 확인만 대기** → ✅ 종결(2026-07-17, 유슨생 결정)

- **이행(유슨생 선승인 2026-07-13, #190 선례 절차)**: **① 유효 수동 비'정상'은 일수 게이트 무관 포함** — 라벨 "○○(수동 지정)"에 일수 사실대로 병기. 수동 "정상" 제외·신규·상담접수 제외(목록 배지 공백 미러)·만료 파생 복귀는 불변. 테스트 4종 잠금(#235 squash `fc3b882`). 선례 정합: #227(재문의 라벨 목록과 통일)·#204(일수 목록과 통일) — 두 번 다 "배지와 AI가 같은 고객".
- **이사님께 묻는 것**: 위 ① 방향 사후 승인(기각이면 필터 두 줄 원복 — 소형). **함께 인지 2건**(답에 따라 박제/후속): ⓐ관리성 쓰기(일괄 담당자 변경 등)도 실활동으로 잡혀 스누즈를 조용히 일괄 만료시킨다(수용이면 박제만) ⓑ수동 상태를 파생으로 되돌리는 해제 UI 부재(저우선 — 필요 시 팝오버에 "자동으로 되돌리기" 옵션).
- **② 확장(2026-07-14, 유슨생 승인·같은 축 — 사후 확인 병기)**: **신규·상담접수 고객도 유효 수동 비'정상'이면 목록 배지 표시 + 리포트 포함**. 유슨생이 실사용에서 신규 고객에 관리 상태를 지정했는데 목록에 반영이 안 되는 것을 버그로 리포트 — 수동 지정 자체가 상담사 액션이라 "액션 전 공백" 규칙의 자기모순이고, 관리 상태 필터·상세 드로어는 이미 수동을 인정해 화면 내 모순이 실존했다(배치 4 감사 B2 기각의 번복 — 기각 근거가 배지 셀만 본 불완전 전제). 수동이 없는 신규·상담접수 공백(파생)은 불변. 이행 PR #240.

### 항목 9 — 계산엔진(솔루션 조회) = 파트너 API 연동 (#241, 2026-07-14): 🟡 **유슨생 주도 확정·머지 완료 — 사후 공유 묶음** → ✅ 종결(2026-07-17, 이사님 컨펌 완료)

원래 "이사님 브레인스토밍 선행" 항목이었으나 유슨생 주도로 설계·구현 완료(스펙 `ref/specs/2026-07-14-crm-solution-quote-integration-design.md` + 개정 1·2). 사후 공유 5건:

1. **계산엔진 = 자체 구축이 아니라 파트너(제프 financial-dolim-solution) API 연동** — 제프 플랫폼이 계산 SSOT. 견적 워크벤치 계산기 버튼이 실동작(금융사 미선택 시 전사 일괄 조회 랭킹 모달 — 제프 UI 이식).
2. **금융사 어휘 = 파트너 8사로 교체**(운용리스 8/장기렌트 3) — 구 목업 어휘(우리금융캐피탈·하나캐피탈)는 저장 견적 표시로만 잔존. 미취급 금융사는 수기 작성(추가 어휘는 한 줄 확장 구조).
3. **금리 의미론 변경**: 금융사 표면금리 → **리스계산기 실질(내재) 금리**(앱 lease_calc RATE 역산 — 예: 표면 5.22% 조건이 실질 7.01%로 표시). 앱 리스계산기와 같은 공식.
4. **결과 4필드(반납/인수/출고 전/금리) = 읽기 전용 자동 파생**(수기 입력 불가) — 구 견적 재진입·재저장 시 과거 수기 값이 파생값으로 대체됨. 산식: 반납=월납입×기간+선수금 / 인수=+잔존가치 / 출고 전=보증금+선수금+기타비용.
5. **월납입 채움 = 제프 표시 규칙**(운용리스 100원 올림) — 원값은 시나리오 스냅샷(solution_raw)에 보존.
