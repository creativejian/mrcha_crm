# 차선생 CRM 고유번호 체계

작성일: 2026-05-18

이 문서는 차선생 CRM에서 사람이 보고 말하고 검색하는 업무용 고유번호 체계를 정의한다.
DB 내부 기본키와 화면/상담/문서에 노출되는 업무 코드는 분리한다.

## 기본 원칙

- 내부 DB 기본키는 `uuid7`을 사용한다.
- 아래 고유번호는 사람이 보는 public/business code다.
- 고객에게 노출될 수 있는 번호는 서비스 규모가 과하게 드러나지 않도록 월 단위 순번을 사용한다.
- 모든 신규 업무 코드는 `PREFIX-YYMM-####` 형식을 기본으로 한다.
- `####`는 해당 월 안에서 1부터 증가하는 4자리 zero-padding 순번이다.
- 한 달에 9,999건을 넘는 것은 현재 차선생 CRM 기준에서 비현실적이므로 5자리로 늘리지 않는다.
- 기존 디엘오토솔루션 데이터도 접수 월 기준을 보존하기 위해 `YYMM`을 포함한다.
- 기존 디엘오토솔루션의 원본 번호는 별도 필드로도 보관한다.

## 확정 코드

| Prefix | 의미 | 대상 | 형식 | 예시 | 카운팅 기준 |
| --- | --- | --- | --- | --- | --- |
| `CU` | Customer | 차선생 고객번호 | `CU-YYMM-####` | `CU-2605-0128` | 월 단위 고객 생성 순번 |
| `DL` | DLAutoSolution Legacy Customer | 기존 디엘오토솔루션 고객번호 | `DL-YYMM-####` | `DL-2605-1420` | 기존 DB 접수 월 기준 |
| `QT` | Quote | 견적번호 | `QT-YYMM-####` | `QT-2605-0342` | 월 단위 견적 발행 순번 |
| `CT` | Contract | 계약번호 | `CT-YYMM-####` | `CT-2605-0048` | 월 단위 계약 생성 순번 |
| `CS` | Consultation | 상담번호 | `CS-YYMM-####` | `CS-2605-0182` | 월 단위 상담 생성 순번 |
| `DV` | Delivery | 출고번호 | `DV-YYMM-####` | `DV-2605-0021` | 월 단위 출고 완료 순번 |
| `ST` | Settlement | 정산번호 | `ST-YYMM-####` | `ST-2605-0017` | 월 단위 정산 생성 순번 |

## 코드별 상세 기준

### 고객번호 `CU`

- 차선생 CRM의 신규 고객에게 발급한다.
- 고객 상세, 상담, 견적, 계약, 검색에서 기본 고객 식별자로 사용한다.
- 고객에게 노출 가능하다.
- 접수일은 별도 `receivedAt` 필드로 보관하므로 코드에는 일자를 넣지 않는다.

예:

```text
CU-2605-0001
CU-2605-0128
CU-2606-0001
```

### 기존 디엘오토솔루션 고객번호 `DL`

- 기존 디엘오토솔루션 CRM에서 가져온 고객 DB의 업무용 참조번호다.
- `DL`은 DLAutoSolution legacy customer를 의미한다.
- 기존 고객 DB가 1만 명 이상이므로 단순 보조값이 아니라 중요한 마이그레이션 기준이다.
- 새 차선생 고객번호 `CU`와 분리한다.
- 기존 원본 번호는 `legacyNo` 또는 `legacyDlNo`로 별도 저장한다.
- 화면 기본 리스트에는 `CU`를 우선 노출하고, 상세/관리자/마이그레이션 맥락에서 `DL`을 노출한다.
- 기존 데이터에도 접수 월이 있으므로 `DL-YYMM-####` 형태로 맞춘다.

예:

```text
DL-2605-1420
```

권장 저장:

```ts
customerCode: "CU-2605-0128"
legacyDlCode: "DL-2605-1420"
legacyDlNo: 11420
legacySource: "DLAutoSolution"
```

### 견적번호 `QT`

- 견적 발행 단위마다 발급한다.
- 같은 고객에게 여러 견적이 발행될 수 있으므로 고객번호와 별도로 관리한다.
- `Quote`가 `Estimate`보다 실제 금융 조건/차량 조건/월 납입금이 담긴 실무 견적에 더 적합하므로 `QT`로 확정한다.
- 제프 견적솔루션과 연결될 주요 업무 코드다.

예:

```text
QT-2605-0342
```

### 계약번호 `CT`

- 계약 생성 시 발급한다.
- 고객 1명에게 계약이 없거나 여러 개 생길 수 있으므로 고객번호와 독립적으로 관리한다.
- 심사, 계약서, 출고, 정산과 연결되는 상위 식별자다.

예:

```text
CT-2605-0048
```

### 상담번호 `CS`

- AI 상담, 상담사 연결, 전화 상담, 재응대 등 상담 세션/이력 단위에 사용한다.
- `CS`는 Consultation으로 정의한다.
- 고객에게 직접 노출하는 번호라기보다 내부 이력 추적용에 가깝다.

예:

```text
CS-2605-0182
```

### 출고번호 `DV`

- 고객이 실제로 차량을 받은 상태가 되었을 때 생성한다.
- 계약 단계에서 미리 생성하지 않는다.
- `Delivery`를 의미하며, `DL`은 디엘오토솔루션 legacy 번호로 쓰기 때문에 출고번호는 `DV`로 확정한다.
- 내부 상태값으로는 release 개념을 별도로 둘 수 있다.

예:

```text
DV-2605-0021
```

권장 상태 흐름:

```text
release_ready    출고 가능/반출 준비
delivery_pending 고객 인도 대기
delivered        고객 인도 완료, DV 발급
```

### 정산번호 `ST`

- 정산 업무가 별도 객체로 생성될 때 발급한다.
- 수수료, 비용, 마진, 입금, 세금계산서 등 정산 흐름을 추적하는 식별자다.

예:

```text
ST-2605-0017
```

## 구현 메모

- 각 prefix와 월별 순번은 중복 없이 발급되어야 한다.
- 동시 생성 가능성을 고려해 DB 레벨 unique constraint와 transaction 또는 sequence table을 사용한다.
- 권장 unique key는 `(prefix, sequenceMonth, sequenceNumber)` 또는 최종 `code` unique다.
- 화면 표시는 `code` 문자열을 그대로 사용한다.
- 검색에서는 하이픈 포함/미포함 모두 대응한다.

예:

```text
CU-2605-0128
CU26050128
2605-0128
0128
```

모두 같은 고객 검색에 활용될 수 있게 정규화 검색을 고려한다.

## 중복 고객 / 중복 접수 메모

- 한 고객은 여러 상담, 여러 견적, 여러 계약, 여러 출고를 가질 수 있다.
- 따라서 고객번호 `CU`와 상담번호 `CS`, 견적번호 `QT`, 계약번호 `CT`, 출고번호 `DV`는 별도 객체의 독립 코드로 관리한다.
- 다만 신규 상담 접수 시 기존 고객인지 확인하는 중복 탐지 UI가 필요하다.
- 중복 탐지는 고유번호 중복과 다른 문제다. `CU`는 한 고객의 기준 식별자이고, 신규 접수 건이 기존 고객에게 연결될 수 있다.
- 이사님은 추후 별도 접수 창을 만들 계획이다. 이때 기존 고객 여부를 명확히 표시하고, 새 고객 생성/기존 고객 연결을 선택할 수 있게 해야 한다.
- 중복 후보 판단 기준은 전화번호를 1순위로 보고, 이름, 고객 유형, 생년월일 또는 사업자 정보, 기존 상담 이력 등을 보조 기준으로 검토한다.
- UI 표현은 나중에 접수 화면 설계 시 확정한다. 예: `기존 고객 가능성`, `기존 고객 연결`, `새 고객으로 접수`.
