# 앱 상담신청 → CRM 통합 설계 노트 (2026-07-07 합의)

> 상태: **방향 합의 완료 · 구현 대기(앱 선행 조건).** 정식 spec/구현은 아래 선행 조건(ⓐⓑⓒ)이
> 앱에서 완료된 뒤 착수한다. 이사님이 "연락처(회원가입 번호 강제) 작업"을 끝내면 알려주기로 함.

## 핵심 비전 (이사님)

**앱 유저 1명 = CRM 고객 1명.** 하나의 앱 유저가 "견적요청"이든 "상담신청"이든, 그 활동 전부가
**하나의 고객 워크벤치(고객 상세)**에 모여 관리되어야 한다.

## 배경 — 앱 유입은 2종

앱에서 CRM으로 들어오는 인바운드는 두 가지다:
1. **견적요청**(`public.quote_requests`) — 기존 파이프라인(#114~#122). `app_user_id`로 CRM 고객에 연결,
   니즈/견적요청 카드로 표시. **전화번호 없음**(카카오 로그인, `profiles.phone_number` 공란).
2. **상담신청**(`public.consultations`) — **현재 CRM에 전혀 안 들어옴**(pending으로 적재만). 견적요청과
   달리 **전화번호를 폼으로 받음**.

## 확인된 사실 (2026-07-07 실측)

- `public.consultations` 컬럼: `id, user_id, customer_name, phone_number, car_model, notes, status, created_at`.
  - 전화번호 O (예: 김어자 010-9876-7820). 관심모델(`car_model`)은 현재 대체로 공란(앱 알림 "관심 모델 미지정").
  - 현재 데이터는 **과도기**: `user_id`가 예약자 단일 계정(대리/폼 입력)이고 `customer_name`/`phone`은 폼 값이라
    실고객과 계정이 불일치. → **앱 회원가입/로그인 강제 후 정상화 전제.**
- 견적요청 승격 로직은 이미 통합 지향:
  - `createCustomerFromRequest`(`src/db/queries/quote-requests.ts:297`)는 **같은 `app_user_id`면 기존 고객을
    반환**(중복 생성 방지, :316) → 한 유저의 여러 요청이 한 고객에 모임.
  - INSERT에 **`phone: profiles.phone_number` 매핑 상주**(:344) → 전화번호가 있으면 CRM 연락처에 자동 반영.
    (지금 앱 유저 9명 전원 공란이라 안 보일 뿐 — 앱이 번호를 받으면 코드 수정 없이 반영됨.)
  - 견적요청 목록은 phone 일치로 기존 고객 매칭도 이미 함(:101,:141, 표시용 read).
- 김민준(수기 고객) 니즈 카드의 **"문의사항" = `crm.customers.need_memo`**(고객 단위 **1개** 필드, 편집 가능).
  편집 폼 저장 필드 = `need_model/need_trim/need_colors/need_method/need_memo`(`useCustomerNeeds.ts`).

## 설계 결정 (합의)

1. **통합 키 = `app_user_id`.** 회원가입 전화번호 강제가 기반(모든 앱 활동이 로그인 계정 기준이 됨).
   상담신청 승격은 견적요청과 **동일 패턴**(app_user_id로 기존 고객 연결, 없으면 생성). `phone`은 CRM 내
   중복 탐지/수기 고객 병합의 보조 키.
2. **상담신청 = 읽기 전용 문의 카드(건별).** 관심모델 + 문의사항(`notes`)을 **원본 그대로** 표시. **수정하지
   않음**(앱 고객이 보낸 원본 = 진실 보존, 견적요청 카드를 편집 못 하게 한 것과 같은 이유).
   - UI는 **김민준 니즈 카드(문의사항 카드) 형태를 건별로 재사용** — 새 디자인 불필요.
   - `need_memo`(고객 단위 1개)에 욱여넣지 않는다(상담신청은 건별 여러 개라 덮어쓰기 문제).
3. **상담사 정리는 별도.** 상담 후 니즈가 구체화되면 `need_memo`(편집 가능) 정리·콜백 일정 등으로 대응.
   원본 문의 카드 위에 상담사 정리가 얹히는 2층 구조.
4. **`source`(상담경로) = 최초 유입 경로 고정.**
   - 견적요청으로 처음 유입 → `앱 견적요청`(기존). 상담신청으로 처음 유입 → **`앱 상담신청`(신규 어휘)**.
   - **이미 한 경로로 유입된 고객이 다른 경로로 활동해도 `source`는 첫 접점 값 유지**, 이후 활동은 카드로 누적.
5. **채팅 아이콘(상담경로 옆 파란 말풍선)은 실시간 채팅용으로 나중에 연결 — 지금은 손대지 않음.**

## 선행 조건 (앱/이사님 몫 — 이게 끝나야 착수)

- ⓐ **앱 회원가입 시 휴대폰번호 입력 강제** — 통합의 기반(상담신청 `user_id`를 신뢰 가능하게).
- ⓑ **상담신청이 로그인 유저 계정으로 들어오게** — 현재 대리/폼 입력 과도기 종료(회원가입 번호로 유입).
- ⓒ **`public.consultations` DDL 계약** — public은 앱 관할(불가침). 견적요청처럼 인계문으로 계약 고정 후
   CRM은 read + 승격만.

## 착수 시 만들 것 (구현 스케치 — 정식 spec 때 확정)

- 상담신청 인바운드: `consultations` → CRM 인박스 수신 → `app_user_id`로 고객 연결/생성(견적요청과 동일),
  `phone→연락처` 매핑, `source` 최초 유입 규칙 적용.
- 고객 상세 워크벤치에 **상담신청 문의 카드**(읽기 전용, 김민준 카드 UI 재사용) 리스트 — 견적요청 카드와 공존.
- 참고 형제 코드: 견적요청 인바운드(`createCustomerFromRequest`·`linkRequestToCustomer`) 패턴 재사용.
