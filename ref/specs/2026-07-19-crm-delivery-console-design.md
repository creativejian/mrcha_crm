## 3. 어휘·스키마 (마이그 0035)

- `client/src/data/customers.ts` `SCHEDULE_TYPE_OPTIONS`: `["재연락","결정확인","체크","견적","안내","요청","내부","심사"]` → 끝에 `"출고"` 추가.
- `src/db/schema.ts`의 `customer_schedules_type_check`는 이 상수 파생 → `db:generate`로 마이그 0035 생성 → `db:migrate` 실 DB 적용(`schemaFilter:["crm"]`, `db:push` 금지).
- additive 변경이라 기존 행 영향 0. **자동 연동(코드 변경 0)**: 고객 상세 드로어 일정 섹션 select에 '출고' 노출, 업무 AI schedule 청크(`날짜(요일) 시간 · 타입 · 메모`)에 타입 값으로 그대로 흐름 — 빌더 텍스트 규칙 불변이라 백필 소급 불필요.

## 4. 서버 — `listCustomers` nextDelivery 파생

- 목록 응답에 **`nextDeliverySchedule: { id, date, time | null } | null`** 동봉: 그 고객의 `customer_schedules` 중 `type='출고' AND done=false`인 행에서 `(scheduled_date, scheduled_time)` 오름차순 첫 행. `id`는 콘솔 팝오버의 수정/삭제 대상 지정에 필수.
- 과거 날짜도 그대로 반환한다(콘솔이 "지남"으로 표시 — 리스크 신호). done=true가 되거나 삭제되면 다음 미완료 일정으로 자연 교체, 없으면 null.
- 비계약완료 고객도 값이 있으면 그대로(소비는 delivery mode만 — 파생을 mode로 왜곡하지 않는다).
- ⚠️ **drizzle 상관 서브쿼리 함정(#154 박제)**: 외부 참조는 `crm.customers.id` 완전정규화 필수 — `${table.col}` 보간이 bare 컬럼으로 렌더링돼 섀도잉되는 실버그 전례. 구현 형태(단일 jsonb 서브쿼리 vs lateral)는 plan에서 확정.
- 그 외 서버 변경 **0**: 단계 전이 = 기존 `PATCH /api/customers/:id`(`statusGroup`/`status` zod + `validateStatusSelection` + CHECK), 일정 생성/수정/삭제 = 기존 일정 CRUD 라우트(embed-on-write 훅 기배선 — '출고' 일정 자동 임베딩·삭제 동기 정리 포함).

## 5. UI 상세

### 5.1 필터 pill (뷰 select 3개 대체)

- delivery mode의 no-op 뷰 select 3개(`담당자별/상담상태별/긴급순`, `NOOP_VIEW_CHANGE`)를 제거하고 **출고 단계 pill**로 교체: `진행 중 N | 딜러사계약중 n | 대리점발주중 n | 특판발주중 n | 배정완료 n | 출고완료 n | 전체 N` (카운트 병기).
- "진행 중" = 출고완료 제외 4단계 합. **기본 선택 = 진행 중**(D8).
- pill은 클라 state(전체보기의 계약가능성/관리상태 필터와 같은 층위 — URL 미편입, `?view` mode만 URL). 다른 mode(consulting/contract/settlement/hold)의 뷰 select는 이번 슬라이스에서 불변.
- #249 SSOT 유지: 레이아웃 클래스는 전 mode 공통 그대로, mode 차이는 필터 항목 선택자 한 곳(`isAllMode` 선택자를 mode별 필터 세트 선택자로 확장)에만.
- **진행 상태 1차/2차 공통 필터는 delivery에서 숨긴다**(구현 중 확정·1440px 실측 — 1차는 계약완료 고정 스코프라 무의미, 2차는 단계 pill과 완전 중복이라 모순 조합(2차=출고완료 ∧ pill=진행 중 → 상시 0명)만 만든다. 잔존 state는 delivery에서 미적용 게이트 — 배치 6 A#3 선례 미러. 담당자·검색은 공통 유지).
- 헤드바 카운트는 현 pill 필터 적용 후 인원 수 기준이며 라벨이 필터명을 반영한다("진행 3명"/"출고완료 4명"/"전체 7명" — "전체 N명" 고정 라벨을 기본 필터에 그대로 쓰면 오독). 로딩 완료 전 숫자 미표시(#287 폴리시 준수).

### 5.2 테이블 컬럼

`선택 | 고객 | 차량 | 출고 단계 | 출고 예정 | 인도 방식 | 담당 | 관리`

- 기존 delivery 컬럼에서 "출고 상태"→"출고 단계"(컨트롤화), "출고 업무"(AI 힌트 평문) 제거(D9), "출고 예정"·"인도 방식" 신설.
- 헤더=바디 정합 테스트 유지(#248 회귀 부류 — heads↔tableColumns 순서 잠금).
- 인도 방식 = `need_delivery_method` 값 그대로(없으면 "—"). 편집은 상세 니즈에서(콘솔 편집 없음).
- 담당·관리(액션 3버튼)·검색·페이지네이션·담당 필터는 콘솔 공통 문법 그대로. 행 클릭 = 상세 드로어(mode 보존, #254).

### 5.3 출고 단계 셀 (전이)

- 전체보기 stage 컨트롤의 **2차 상태 버튼+팝오버 재사용**(1차 버튼·화살표 미렌더). 팝오버 옵션 = 계약완료 종속 2차 5종만.
- 계약완료 그룹 이탈(계약취소 등 1차 변경)은 이 큐의 몫이 아님 — 전체보기/상세에서(의도, 각주).
- 전이 성공 시 기존 워크플로우 낙관 갱신 문법(row 갱신·`markFinalUpdate`·chance 확정 동기 `buildWorkflowPatch`) 그대로. pill 카운트는 row 파생이라 자동 갱신.
- 출고완료 전이 시: 기본 필터(진행 중)에서는 행이 필터 밖으로 빠진다(업무함 소진 의미론). **일정 자동 done 처리 없음**(§5.4 결합 없음 원칙).

### 5.4 출고 예정 셀 (표시 + 팝오버)

- 표시: 값 있음 = `M/D (요일) HH:mm`(시간 없으면 날짜만) / 없음 = `+ 미지정` 버튼형 / **미완료인데 날짜가 KST 오늘 이전 = 붉은 "지남" 배지**(기존 `kst-date.ts` 헬퍼 재사용 — 브라우저 로컬 tz 금지, #204 파리티 부류).
- 클릭 → 팝오버: 날짜(date input, 필수)·시간(선택)·저장·삭제(기존 일정 있을 때만)·취소. 외부 클릭 닫기 등은 기존 목록 팝오버 문법(#216).
- 저장 의미론: `nextDeliverySchedule` 없으면 **생성**(`type:'출고'`, `done:false`, memo 빈 값), 있으면 그 `id`를 **PATCH**(날짜/시간만). 삭제 = 그 id DELETE.
- **복수 '출고' 일정 엣지**: 표시·편집 대상은 항상 서버가 준 earliest 미완료 1건. 나머지는 상세 일정 섹션에서 관리(콘솔은 대표 1건 통로 — 의도, 각주).
- **결합 없음 원칙**: 전이↔일정은 독립. 출고완료 후 남은 미완료 '출고' 일정은 "지남"/상세에서 정리 유도(자동 조작으로 데이터가 몰래 바뀌는 경로를 만들지 않는다).

### 5.5 정렬·빈 상태

- 기본 정렬: `nextDeliverySchedule` 날짜 오름차순, **미지정은 뒤**, 같은 날짜 안에서 시간 오름차순·시간 미지정은 그 날짜의 뒤, 동률·미지정 그룹 내부는 기존 목록 정렬 유지.
- 빈 상태: pill 필터 결과 0명 문구(기존 콘솔 빈 상태 문법). 검색·담당 필터와 AND 결합.
